# Go 编码规范

> 来源：[Google Go Style Guide](https://google.github.io/styleguide/go/guide)

## 优先级原则

- 清晰度 > 简洁性 > 简单性 > 可维护性 > 一致性
- 代码的目的和原理对读者必须清晰，不需要预备知识
- 代码的读者体验优先于编写体验

## 命名规范

- 用 `MixedCaps` 或 `mixedCaps`（驼峰），不用蛇形 `snake_case`
- 导出标识符大写开头（`MaxLength`）；未导出标识符小写开头（`maxLength`）
- 命名不在使用时产生冗余；考虑上下文，避免重复已明确的概念
- 局部变量遵循未导出命名规则（小写开头）

## 格式

- 所有源文件必须符合 `gofmt` 工具输出格式
- 无固定行长限制；优先重构代码而非强制折行
- 不在缩进变化前折行（如函数声明、条件语句）

## 注释

- 注释解释「为什么」，不解释「做什么」
- 不写冗余注释，让代码自说明
- 不常见模式必须加注释，说明原因（通常与性能相关）
- 复杂代码必须配文档说明其目的

## 错误处理

- 标准错误检查模式：`if err := doSomething(); err != nil {}`
- 逻辑与常见模式相反时，用注释突出（如 `// if NO error`）
- 使用有意义的错误信息帮助维护者定位问题

## 依赖与抽象

- 优先用核心语言构造（channel、slice、map、循环、struct）
- 找不到时考虑标准库；最后才引入新依赖或自定义实现
- 最小化隐式和显式依赖；减少包依赖 = 减少影响行为的代码
- 避免不必要的抽象层
- 接口会移除上下文信息，收益必须抵消成本才引入
- 接口文档必须说明调用者需了解的实现细节

## 代码组织

- 代码从上到下易于阅读，不假设读者已知其内容
- 隐藏关键逻辑的辅助函数会增加维护负担
- 可预测的命名便于用户和维护者使用

## 测试

- 使用全面的测试套件确保行为和逻辑正确
- 测试提供清晰、可操作的诊断信息
- 可运行 example 演示正确用法（复杂代码尤其需要）
- 表格驱动测试适合分解重复测试逻辑

## 维护性

- 代码修改次数远多于编写次数，优先考虑易于修改
- API 能优雅扩展；避免过早加入未使用功能
- 选择映射到问题域而非代码结构的抽象

## 一致性

- 一致的代码在外观、感觉、行为上与更广泛代码库相似
- 同一概念用同一名称，便于理解和重构
- 包内一致性最重要；避免同一问题多种处理方式
- 风格指南无规定时，遵循临近代码的一致选择
- 全局一致性高于本地一致性；不因本地风格违反文档化规范
