# Shell/Bash 编码规范

> 来源：[Google Shell Style Guide](https://google.github.io/styleguide/shellguide.html)

## 基础规范

- 可执行脚本用 bash：起始行 `#!/bin/bash`
- 脚本超过 100 行或逻辑复杂时改用结构化编程语言（Python 等）
- 文件头包含内容描述注释

## 命名规范

- 函数名：小写加下划线，包文件用 `::` 分隔（如 `mypackage::func`）
- 变量名：小写加下划线
- 常量和环境变量：大写加下划线，用 `readonly` 声明
- 源文件名：小写加下划线，无需特定扩展名

## 格式约定

- 缩进 2 空格，禁用制表符（here 文档的 `<<-` 除外）
- 行长限制 80 字符；超长字符串用 here 文档或嵌入换行
- 管道链若无法单行，按每个管道符一行格式排列
- `if`/`for`/`while` 的 `then`/`do` 与关键字同行；`else`/`fi`/`done` 独占行

## 变量引用

- 用 `${var}` 而非 `$var`（`$1`、`$?` 等特殊变量除外）
- 含变量、命令替换、空格或元字符的字符串必须加双引号
- 列表参数存入数组，展开时用 `"${array[@]}"` 保留空格
- 字符串比较用 `==`；数值比较用 `(( ))` 或 `-lt/-gt`

## 安全与错误处理

- 错误消息输出到 STDERR：用 `>&2` 重定向
- 必须检查所有命令的返回值；用 `if` 或 `$?` 判断
- 管道返回值用 `PIPESTATUS` 数组立即检查（赋值前先保存）
- 禁止在 shell 脚本中用 SUID/SGID；需提权用 `sudo`

## 条件测试

- 优先用 `[[ ]]` 而非 `[ ]` 或 `test`（支持正则匹配，无路径展开问题）
- 判断空字符串用 `-z/-n`，不用 filler 字符技巧
- 禁用 `eval`；难以跟踪变量赋值且存在注入风险
- 不用 `alias`；用函数替代以获得更好的参数处理

## 高级特性

- 命令替换用 `$(command)` 而非反引号，便于嵌套
- 不要把管道输出传给 `while` 循环（子 shell 导致变量丢失）；用进程替换或 `readarray`
- 数值运算用 `(( ))` 或 `$(( ))`；禁用 `let`、`expr` 或 `$[ ]`
- 用 ShellCheck 识别常见 bug 和警告

## 函数与代码组织

- 函数集中在文件开头（常量下方）；可执行代码不穿插在函数间
- 函数内局部变量必须用 `local` 声明，避免全局污染
- 命令替换的返回值检查需单独语句（不能合并声明和赋值）
- 长脚本用 `main()` 函数作入口，文件末尾调用 `main "$@"`

## 输入输出

- 不用管道读文件到 `while`；用 `while read < <(command)` 替代
- 读输出到数组用 `readarray`（bash 4+），同样需进程替换
- 传递参数用 `"$@"` 而非 `$*`，前者保留参数边界

## 特殊情况

- 通配符展开用 `./filename` 而非 `filename`，防止文件名以 `-` 开头误解
- 可执行文件用 `.sh` 扩展名或无扩展名；库文件必须有 `.sh` 且不可执行
- TODO 注释格式：`# TODO(name): 描述`
- 优先用 bash 内建命令而非外部工具（如用参数展开代替 `sed`）
- 规则冲突时保持项目既有风格一致
